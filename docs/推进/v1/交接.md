## 3. 当前代码结构说明

### `source/server/server.mjs`
职责：

- 提供本地 HTTP 服务
- 提供 API：
  - `GET /api/health`
  - `GET /api/project/current`
  - `POST /api/project/select`
- 提供静态页面服务，直接把 `source/web` 作为前端资源目录

### `source/server/lib/project-store.mjs`
职责：

- 维护当前项目状态的本地读写
- 当前保存位置：`source/server/data/project-state.json`
- 该目录已加入 `.gitignore`

### `source/server/lib/folder-dialog.mjs`
职责：

- 调用 Windows PowerShell 的 `System.Windows.Forms.FolderBrowserDialog`
- 让后端弹出原生文件夹选择窗口

注意：

- 这部分是 **Windows-only** 的本地实现
- 当前完全符合项目目前的本地 Unity 开发场景

### `source/web/index.html`
职责：

- M1 页面骨架
- 展示当前项目、服务状态和状态输出区

### `source/web/app.js`

### `source/web/styles.css`
职责：

## 4. 当前 API 说明

### `GET /api/health`
返回服务健康状态。

示例响应：

```json
{
  "ok": true,
  "app": "SimpleKanban",
  "version": "0.1.0-m1"
}
```

### `GET /api/project/current`
返回当前保存的项目路径。

示例响应：

```json
{
  "project": {
    "name": "SimpleKanban",
    "path": "D:\\Projects\\SimpleKanban",
    "updatedAt": "2026-03-07T01:00:53.224Z"
  }
}
```

### `POST /api/project/select`
用途有两种：

#### 模式 A：网页正常调用
请求体为空对象 `{}` 时：

- 服务端弹出原生文件夹选择器
- 用户选择后保存项目路径

#### 模式 B：脚本 / 测试调用
请求体可传：

```json
{
  "path": "D:\\Projects\\SimpleKanban"
}
```

这样可以跳过原生对话框，直接设置项目路径。

这个能力是为了：

- 自动化测试方便
- 调试方便
- 后续接口复用更方便

## 7. 当前设计取舍

### 为什么 M1 没直接上 React / Vite
因为当前阶段最重要的是：

- 尽快打通本地服务
- 确认项目选择链路可行
- 确认前后端连接方式可行
- 为 M2 的 PTY / 网页终端留出清晰落点

如果一开始就上完整前端栈，会增加很多与当前目标无关的初始化成本。

### 为什么项目选择放后端
原因：

- 纯浏览器不适合稳定获取本地绝对路径
- 当前能力本质上是本地系统能力，不是 Web 沙盒能力
- 后端弹原生目录选择器更符合后续“本地工具”定位

### 为什么保留 `path` 直传模式
因为它能明显降低调试和测试成本，也方便后续 TaskCard 或执行器模型复用这条接口。

## 9. 下一步建议（直接进入 M2）

建议新对话接手后直接开始 `M2：网页终端最小闭环`。

推荐推进顺序：

1. 引入 `xterm.js`
2. 接入 `node-pty`
3. 用 WebSocket 打通终端输出
4. 打通前端输入写回 PTY
5. 支持终端 `resize`
6. 设计最小 `TerminalSession` 快照模型

## 10. M2 开始前建议保留的约束

- 继续保持 **单机 / 单项目 / 单活跃终端** 的最小范围
- 不要提前引入 TaskCard 业务逻辑
- 不要提前引入账号 / 协作 / 远程服务
- 不要提前引入 worktree
- 外部 terminal 继续保持为后续扩展，不作为当前主路径
