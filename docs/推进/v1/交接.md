## 4. 当前 API 说明

### `GET /api/health`
返回服务健康状态。

示例响应：

```json
{
  "ok": true,
  "app": "SimpleKanban",
  "version": "0.1.0-m1"
}
```

### `GET /api/project/current`
返回当前保存的项目路径。

示例响应：

```json
{
  "project": {
    "name": "SimpleKanban",
    "path": "D:\\Projects\\SimpleKanban",
    "updatedAt": "2026-03-07T01:00:53.224Z"
  }
}
```

### `POST /api/project/select`
用途有两种：

#### 模式 A：网页正常调用
请求体为空对象 `{}` 时：

- 服务端弹出原生文件夹选择器
- 用户选择后保存项目路径

#### 模式 B：脚本 / 测试调用
请求体可传：

```json
{
  "path": "D:\\Projects\\SimpleKanban"
}
```

这样可以跳过原生对话框，直接设置项目路径。

这个能力是为了：

- 自动化测试方便
- 调试方便
- 后续接口复用更方便

## 7. 当前设计取舍

### 为什么 M1 没直接上 React / Vite
因为当前阶段最重要的是：

- 尽快打通本地服务
- 确认项目选择链路可行
- 确认前后端连接方式可行
- 为 M2 的 PTY / 网页终端留出清晰落点

如果一开始就上完整前端栈，会增加很多与当前目标无关的初始化成本。

### 为什么项目选择放后端
原因：

- 纯浏览器不适合稳定获取本地绝对路径
- 当前能力本质上是本地系统能力，不是 Web 沙盒能力
- 后端弹原生目录选择器更符合后续“本地工具”定位

### 为什么保留 `path` 直传模式
因为它能明显降低调试和测试成本，也方便后续 TaskCard 或执行器模型复用这条接口。
